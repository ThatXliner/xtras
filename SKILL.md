# Writing Commit Messages

## Core Principle

Commit msg = for person reading `git log` 6 months later. Tell **why** change exists — diff shows *what*/*how*.

## Pre-Commit Checklist

1. **Only stage relevant files** — run `git status` and `git diff` to review the working tree. Never use `git add -A` or `git add .`. Stage files selectively:
   - **If there's prior conversation context:** only `git add` files that were created or modified as part of the current task. Leave pre-existing unstaged/untracked changes alone — they are the user's in-progress work.
   - **If invoked with no prior context (e.g. user just ran `/commit`):** you may stage all pending changes, **unless** some changes look like incomplete features (half-written code, TODO stubs, broken imports). In that case, warn the user and only commit what looks complete. If the user insists (e.g. "go ahead, I know what I'm doing"), commit everything they ask for.
2. **Run lints** — all linters must pass. Do not `git commit --no-verify`.
3. **Update documentation** — if the code change should reflect updates in any docs or architecture files, update them first and include in the commit. If the documentation changes would be too big (e.g. complicated "whys", etc) create a separate commit about updating documentation.
4. **NEVER commit AI-generated plan files** (e.g., design docs generated by Superpowers typically in `docs/superpowers/` or similar agent workflows). These are ephemeral agent guidance, not proper project documentation.
5. **Review staged changes** for atomicity (see below).

## Atomic Commits

**Default to splitting** — granular commits make `git revert`, `git bisect`, and code review easier.

**The splitting criterion is motivation, not subsystem.** Split when changes have different *whys*, not when they touch different files or categories. "I was fixing X and noticed unrelated Y" = two commits. "I built X which required touching A, B, and C" = one commit. "Add crash recovery and its error reporting" = one commit (same why). "Fix typo and add feature" = two commits (different whys).

**Don't merge because of code proximity.** Two features that touch the same function or file are still two commits if they solve different problems. Interleaved changes (where feature A and feature B both modify the same block) should still be split — use `git add -p` to stage hunks selectively, or commit one feature first, then the other. Common trap: "they share infrastructure / touch the same loop" is not a single why.

TL;DR: if you are to create a single commit, you must have an incredibly good reason for doing so. Even splitting the commit to little parts of the subsystem is better than a single commit that touches everything.

## Gitmoji + Conventional Commits

**Format:** `:gitmoji: type(scope): imperative description`

Use GitHub shortcode syntax combined with Conventional Commits. Prefer specific gitmojis over general ones (e.g. `:passport_control:` for login logic, not `:sparkles:` or `:rocket:` not `:wrench:`).

### Gitmoji Reference

Match the gitmoji to the conventional commit `type` semantically. Prefer specific over general (e.g. `:ambulance:` over `:bug:` for critical hotfixes, `:passport_control:` over `:sparkles:` for a new login page).

| Gitmoji | Use for |
|---------|---------|
| :sparkles: | New feature |
| :bug: | Bug fix |
| :ambulance: | Critical hotfix |
| :recycle: | Refactor code |
| :fire: | Remove code/files |
| :zap: | Performance |
| :memo: | Documentation |
| :white_check_mark: | Tests |
| :arrow_up: / :arrow_down: | Upgrade/downgrade deps |
| :lock: | Security fix |
| :boom: | Breaking changes |
| :pencil2: | Fix typos |
| :lipstick: | UI/style |
| :art: | Code style |
| :wrench: | Config files |
| :construction_worker: | CI/build |
| :truck: | Move/rename files |

For niche gitmojis (:passport_control:, :thread:, :card_file_box:, etc.), see https://gitmoji.dev/

### Type Pairing

The gitmoji and conventional commit `type` are independent but related. The gitmoji can be more specific than the type. E.g. `:ambulance: fix(auth):` uses the general `fix` type but the specific `:ambulance:` emoji.

## Subject Line Rules

1. **50 characters max** (hard limit: 72, not counting gitmoji shortcode)
2. **Imperative mood** — "add", "fix", "remove", not "added", "fixes", "removed"
3. **Lowercase after colon** — `:bug: fix(auth): handle null user`
4. **No period at end**
5. **Say why, not what** — the diff shows what changed. If truly self-explanatory (e.g., `:pencil2: fix(docs): correct "recieve" → "receive"`), "what" is acceptable.

```
# BAD: says what
:bug: fix(auth): add null check before accessing user.email
# GOOD: says why
:bug: fix(auth): prevent crash when user session expires

# BAD: wrong gitmoji and type
:sparkles: feat(deps): upgrade React to 19.0
# GOOD
:arrow_up: build(deps): upgrade React to 19.0
```

## Body (When Needed)

Add a body when the subject alone doesn't explain **why**. Include for bug fixes, non-obvious decisions, breaking changes, or wide blast radius. Skip for self-explanatory changes.

- Blank line between subject and body
- Explain motivation and context, not a play-by-play of the diff

```
:bug: fix(orders): prevent duplicate submissions on double-click

Users in production were creating duplicate orders by clicking Submit
before the first request completed. Debouncing alone wasn't sufficient
because slow network conditions extended the vulnerable window.

Disable the button on first click and re-enable on error. Chose this
over request deduplication to give users immediate visual feedback.
```

## Running the Commit

**Never use heredocs (`cat <<'EOF'`) for commit messages.** The Bash tool's quoting interacts badly with heredocs and causes syntax errors, wasting tokens on retries. Instead, pass the message directly with `git commit -m "subject\n\nbody"`.
